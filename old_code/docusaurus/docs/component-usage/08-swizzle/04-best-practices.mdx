# Swizzling Best Practices

Follow these guidelines to maintain a stable and maintainable documentation site.

## Do's and Don'ts

### ✅ Do's

1. **Start with wrapping**
   ```jsx
   import OriginalComponent from '@theme-original/ComponentName';
   
   export default function ComponentWrapper(props) {
     // Add your customizations
     return <OriginalComponent {...props} />;
   }
   ```

2. **Document your swizzles**
   - Keep a list of swizzled components
   - Document why each was swizzled
   - Note any custom behavior added

3. **Use TypeScript when possible**
   ```bash
   npm run swizzle @docusaurus/theme-classic ComponentName -- --typescript
   ```

4. **Test after Docusaurus updates**
   - Swizzled components may break
   - Keep original component references
   - Have a rollback plan

5. **Minimize global CSS**
   - Prefer CSS modules when possible
   - Scope global CSS carefully
   - Use CSS variables for theming

### ❌ Don'ts

1. **Don't swizzle internal components**
   - Components marked as "Forbidden"
   - Undocumented components
   - Implementation details

2. **Don't over-swizzle**
   - Only swizzle what you need
   - Consider alternatives first
   - Each swizzle increases maintenance

3. **Don't ignore safety warnings**
   ```bash
   # Check safety level first
   npm run swizzle @docusaurus/theme-classic -- --list
   ```

## Component Organization

### File Structure
```
src/theme/
├── ComponentName/
│   ├── index.js          # Main component
│   ├── styles.module.css # Component styles
│   └── utils.js          # Helper functions
└── theme.d.ts            # TypeScript definitions
```

### Naming Conventions
- Match Docusaurus's component names exactly
- Use PascalCase for components
- Use camelCase for utilities
- Use kebab-case for CSS classes

## CSS Strategies

### 1. CSS Modules (Preferred)
```jsx
import styles from './styles.module.css';

export default function MyComponent() {
  return <div className={styles.myClass}>Content</div>;
}
```

### 2. Global CSS (When Necessary)
```css
/* src/css/component-overrides.css */
/* Use specific selectors to avoid conflicts */
.theme-doc-sidebar .menu__link--active {
  background: var(--custom-color);
}
```

### 3. CSS-in-JS
```jsx
const StyledComponent = styled.div`
  background: ${props => props.theme.primary};
`;
```

```json
// package.json
{

    "@docusaurus/core": "3.8.1",
```

### Testing Checklist
- [ ] Component renders correctly
- [ ] Styles apply as expected
- [ ] Mobile responsive
- [ ] Dark mode compatible
- [ ] Accessibility maintained
- [ ] Performance acceptable

### Update Process
1. Read Docusaurus changelog
2. Test in development
3. Check swizzled components
4. Update if necessary
5. Test thoroughly
6. Deploy carefully

## Common Patterns

### Adding Analytics
```jsx
// src/theme/Root/index.js
import {useEffect} from 'react';

export default function Root({children}) {
  useEffect(() => {
    // Initialize analytics
  }, []);
  
  return children;
}
```

### Custom Search
```jsx
// src/theme/SearchBar/index.js
export default function SearchBar() {
  return <CustomSearchImplementation />;
}
```

### Theme Extensions
```jsx
// src/theme/Layout/index.js
import Layout from '@theme-original/Layout';
import {ThemeProvider} from './ThemeContext';

export default function LayoutWrapper(props) {
  return (
    <ThemeProvider>
      <Layout {...props} />
    </ThemeProvider>
  );
}
```

## Debugging Tips

### Component Not Loading
1. Check file location
2. Verify component name
3. Clear Docusaurus cache
4. Check for import errors

### Styles Not Applying
1. Check CSS specificity
2. Verify import order
3. Use browser DevTools
4. Consider global CSS

### Build Failures
1. Check for TypeScript errors
2. Verify all imports
3. Test without swizzles
4. Check Docusaurus version
