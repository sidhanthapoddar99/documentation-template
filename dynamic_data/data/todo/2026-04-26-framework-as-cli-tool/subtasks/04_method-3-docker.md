Method 3: run or deploy the framework via Docker. A Claude skill prompts the user for their scenario (dev-in-container vs production-deploy, plus topology / mode / domain) and generates `Dockerfile` + `docker-compose.yml` + `nginx.conf`. The same artifacts also serve as a documented hand-rollable pattern for users who'd rather copy-paste than run a skill.

## Design reference

> Detailed design lives in **[`notes/01_docker-deployment-design.md`](../notes/01_docker-deployment-design.md)** — read that for the architecture: the two roles Docker plays (running vs deploying), all four topologies (subpath single-nginx / subpath multi-nginx / subdomain / standalone), multi-stage Dockerfile pattern, docker-compose shapes per topology, nginx config snippets, volume mount strategy, mode-to-container mapping, and what the skill prompts/produces.
>
> This subtask only holds the work checklist; design decisions live in the note.

## Out of scope (deliberately dropped from the deployment issue, see note for reasoning)

- **Platform-specific deployment guides** (Vercel, Netlify, GH Pages, Cloudflare Pages) — downstream of the framework; not Method-3-specific. Could be filed as a separate issue if demand surfaces.
- **CI/CD templates** (GitHub Actions, GitLab CI) — same reasoning. The static build is the artifact; how to push it is platform-specific.
- **Content-type creation skills** (issues, blogs, etc., from the deployment issue's subtask 02) — already covered by the existing `documentation-guide` plugin.
- **Kubernetes manifests** — docker-compose only for v1; users can `kompose convert` if needed.

## Open questions (design-level — decide before building the skill)

1. Where does `mode: build` write `dist/` — internal to the container (multi-stage build copies to nginx stage), or mounted volume (host gets the artifact for further processing)? Note recommends "internal" as default with mounted as opt-in for CI.
2. Single Dockerfile with two targets vs two separate Dockerfiles for Role 1 / Role 2? Ergonomics call.
3. Does the deployment skill ship inside the existing `documentation-guide` plugin, or as a separate plugin? Probably the former — same delivery story.
