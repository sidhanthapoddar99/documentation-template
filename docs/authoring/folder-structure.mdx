# Folder Structure

Understanding folder structure is key to creating well-organized documentation. This guide explains how folders and files are interpreted by the framework.

## Basic Structure

```
docs/
├── section-name/              # A documentation section
│   ├── index.mdx              # Section landing page
│   ├── page-one.mdx           # Child page
│   ├── page-two.mdx           # Child page
│   └── assets/                # Assets (excluded from sidebar)
│       ├── example.ts
│       └── diagram.png
├── another-section/
│   └── ...
└── standalone.mdx             # Top-level page
```

## Folder Types

### 1. Folder with Index (Clickable Section)

When a folder contains an `index.md` or `index.mdx` file, it becomes a **clickable section** in the sidebar.

```
components/
├── index.mdx          # ← This makes "Components" clickable
├── buttons.mdx
├── cards.mdx
└── forms.mdx
```

**Sidebar appears as:**
```
▼ Components          ← Clickable, links to index.mdx
    Buttons
    Cards
    Forms
```

The index file typically contains:
- An overview of the section
- Links to child pages
- Summary of what's covered

```markdown
---
title: Components
description: UI component documentation
sidebar_position: 1
---

# Components

This section covers all available UI components.

## Available Components

- [Buttons](/docs/components/buttons) - Button variants and usage
- [Cards](/docs/components/cards) - Card layouts
- [Forms](/docs/components/forms) - Form elements
```

### 2. Folder without Index (Non-Clickable Group)

When a folder has **no index file**, it becomes a non-clickable category label.

```
api-reference/
├── authentication.mdx
├── endpoints.mdx
└── errors.mdx
```

**Sidebar appears as:**
```
▾ Api Reference       ← Not clickable, just a label
    Authentication
    Endpoints
    Errors
```

Use this pattern when:
- The section doesn't need an overview page
- You just want to group related pages visually
- The first child page serves as the introduction

## Special Folders

### Assets Folder

Any folder named `assets` is **completely excluded** from the sidebar and routing:

```
configuration/
├── index.mdx
├── themes.mdx
└── assets/           # ← Excluded from sidebar
    ├── theme-example.css
    ├── config-sample.json
    └── screenshot.png
```

Use assets folders for:
- Code snippets to import into MDX
- Images and diagrams
- Configuration examples
- Any supporting files

See [Using Assets](/docs/authoring/assets) for details on importing these files.

## File Naming Conventions

### Allowed Characters

- Lowercase letters: `a-z`
- Numbers: `0-9`
- Hyphens: `-` (preferred for word separation)
- Underscores: `_` (alternative to hyphens)

### Naming Best Practices

| Good | Avoid |
|------|-------|
| `getting-started.mdx` | `Getting Started.mdx` |
| `api-reference.mdx` | `API Reference.mdx` |
| `01-introduction.mdx` | `1. Introduction.mdx` |

### URL Generation

Filenames become URL segments:

| File | URL |
|------|-----|
| `docs/getting-started/index.mdx` | `/docs/getting-started` |
| `docs/getting-started/installation.mdx` | `/docs/getting-started/installation` |
| `docs/api/endpoints.mdx` | `/docs/api/endpoints` |

## Ordering Pages

### Using sidebar_position

The most reliable way to order pages:

```markdown
---
title: Introduction
sidebar_position: 1
---
```

```markdown
---
title: Installation
sidebar_position: 2
---
```

```markdown
---
title: Configuration
sidebar_position: 3
---
```

### Using Filename Prefixes

Alternative approach using numeric prefixes:

```
getting-started/
├── 01-introduction.mdx
├── 02-installation.mdx
├── 03-configuration.mdx
└── 04-first-steps.mdx
```

Note: `sidebar_position` always takes precedence over filename ordering.

### Mixed Ordering

When some pages have `sidebar_position` and others don't:

1. Pages with `sidebar_position` appear first, in numeric order
2. Pages without `sidebar_position` appear after, in alphabetical order

## Nesting Depth

You can nest folders multiple levels deep:

```
docs/
└── platform/
    └── features/
        └── authentication/
            ├── index.mdx
            ├── oauth.mdx
            └── jwt.mdx
```

URL: `/docs/platform/features/authentication/oauth`

**Recommendation:** Keep nesting to 2-3 levels maximum for better navigation.

## Example: Complete Documentation Structure

```
docs/
├── index.mdx                      # Docs home
├── getting-started/
│   ├── index.mdx                  # sidebar_position: 1
│   ├── installation.mdx           # sidebar_position: 2
│   └── quick-start.mdx            # sidebar_position: 3
├── guides/                        # No index (non-clickable)
│   ├── basic-usage.mdx
│   ├── advanced-topics.mdx
│   └── assets/
│       └── example-config.json
├── components/
│   ├── index.mdx
│   ├── buttons.mdx
│   ├── cards.mdx
│   └── assets/
│       ├── button-example.tsx
│       └── card-example.tsx
├── api-reference/
│   ├── index.mdx
│   ├── authentication.mdx
│   ├── endpoints.mdx
│   └── errors.mdx
└── changelog.mdx                  # Standalone page
```

## Summary

| Pattern | Clickable? | Use When |
|---------|-----------|----------|
| Folder with `index.mdx` | Yes | Section needs an overview page |
| Folder without index | No | Just grouping related pages |
| `assets/` folder | N/A | Storing supporting files |
| Top-level `.mdx` file | Yes | Standalone page |
