
# Project Overview

This is a Docusaurus-based documentation site for NeuraLabs with a custom theme system and component architecture.

# Important Instructions


## 1. Component Usage Documentation
- **Location**: `/docs/component-usage/<componet_name>.mdx`\
- `/docs/component-usage/index.mdx` should contian breif about all components
- **Purpose**: This file documents how to use all custom elements and components
- **Important**: Always update this file when creating new components or elements
- **Component location**: `/src/components/elements/`

## 2. Color Theme System

The project uses a centralized color management system:

1. **Color Definitions**: `src/theme/colors.js`
   - All colors are defined here
   - Supports light/dark mode variants
   - Includes semantic color mappings
   - Icon filters for SVG images

2. **Color Generation**:
   ```bash
   node src/theme/generateColors.js
   ```
   - Run this after modifying colors.js
   - Generates `src/css/generated-colors.css`
   - DO NOT manually edit generated-colors.css

3. **Color Usage**:
   - Use semantic variables: `var(--theme-bg-primary)`, `var(--theme-text-primary)`
   - Brand colors: `var(--color-brand-primary)`
   - All defined in generated-colors.css



## 3. Routing Configuration

Each documentation section has its own plugin instance:
- Configured in `docusaurus.config.js`
- Uses `@docusaurus/plugin-content-docs`
- Autogenerated sidebars from folder structure
- Each route is independent with its own path


**Important Notes:**
- Base plugin directories do NOT need `_category_.json` files
- Direct subdirectories of base paths should be Level 1 categories with `sidebar-header` class
- Check `docusaurus.config.js` when adding new sections

## 4. Component Development Workflow

When creating new components:

1. **Create component** in `/src/components/elements/ComponentName/`
   - ComponentName.js
   - ComponentName.css
   - index.js (export)

2. **Document the component** in `/docs/component-usage/index.mdx`
   - Add usage examples
   - Show all props/options
   - Include code snippets

3. **Use theme colors** in CSS:
   ```css
   .my-component {
     background: var(--theme-bg-primary);
     color: var(--theme-text-primary);
     border: 1px solid var(--theme-border-default);
   }
   ```

## 5. SVG Icons

- All SVG icons use pure black (#000000) strokes
- Dark mode automatically inverts them using `filter: invert(1)`
- Controlled by `--theme-icon-filter` CSS variable

## 9. Sidebar UX Pattern

The documentation implements a clean sidebar hierarchy:

1. **Base Plugin Directories** (No _category_.json):
   - These are defined in `docusaurus.config.js` as plugin paths
   - Each represents a separate doc section with its own route
   - Example: `docs/overview`, `docs/platform/neuralock`

2. **Level 1 Categories** (Direct children of base dirs):
   - Non-clickable headers with bold text
   - Must use `className: "sidebar-header"` and `collapsible: false`
   - Act as organizational labels only
   - Use title case (first letter capitalized, not all caps)
   - Example:
   ```json
   {
     "label": "Components",
     "position": 1,
     "className": "sidebar-header",
     "collapsible": false
   }
   ```

3. **Level 2+ Categories**:
   - Normal navigable items
   - Should link to their first document (no generated index)
   - **Doc IDs must be relative to the plugin's base path**
   - Example:
   ```json
   {
     "label": "Cards & Containers",
     "position": 1,
     "link": {
       "type": "doc",
       "id": "cards-containers/card"  // Relative to plugin base, not full path
     }
   }
   ```

**Important**: The custom sidebar CSS (`src/css/custom-sidebar.css`) handles the visual styling for this pattern.

## 10. Documentation Writing Guidelines

### Asset Management:
1. **Mermaid Diagrams**: 
   - Create `.mermaid` files in `assets/` folder
   - Use `MermaidDiagram` component with file import
   - Example:
   ```jsx
   import FlowchartSource from '!!raw-loader!./assets/flowchart.mermaid';
   <MermaidDiagram content={FlowchartSource} filename="flowchart.mermaid" />
   ```

2. **Graphviz Diagrams**:
   - Create `.dot` files in `assets/` folder
   - Use appropriate component with file import

3. **Code Snippets**:
   - For all code create separate files in `assets/`
   - Use `FileCollapsibleCodeBlock` and file imports
   - Example:
   ```jsx
   import ExampleCode from '!!raw-loader!./assets/<mdx_filename>/example.js';
   <FileCollapsibleCodeBlock content={ExampleCode} filename="example.js" />
   ```
   - dont use code directly in MDX, always use file imports

## 11. Parallel Documentation Creation:
When creating large documentation sections, use parallel workers:
1. Each worker should read all relevant source files
2. Workers should understand the complete MDX component system
3. Coordinate to avoid duplication
4. Follow consistent formatting and structure

## 12. Common Tasks

### Adding a new component:
1. Create in `/src/components/elements/NewComponent/`
2. Document in `/docs/component-usage/index.mdx`
3. Use theme colors from CSS variables

### Modifying colors:
1. Edit `src/theme/colors.js`
2. Run `node src/theme/generateColors.js`
3. Colors automatically update across the site

### Adding a new doc section:
1. Create folder in `/docs/`
2. Add MDX files
3. Create assets folder for diagrams and code
4. Sidebar auto-generates

### Creating a custom page:
1. Add `.js` file to `/src/pages/`
2. Use existing layouts from `/src/layouts/`
3. Import components from `/src/components/elements/`

## 13. Correct Import Patterns

### Standard Imports (Always include these in every MDX file):
```jsx
// Custom Components
import { Card, CardHeader, CardTitle, CardDescription } from '@site/src/components/elements/Card';
import { Callout } from '@site/src/components/elements/Callout';
import { CollapsibleCodeBlock, InlineCodeCard, FileCollapsibleCodeBlock, FileInlineCodeCard } from '@site/src/components/elements/CodeBlock';
import { Features, Feature } from '@site/src/components/elements/Features';
import { CustomMermaid, MermaidDiagram } from '@site/src/components/elements';
import { GraphViz, GraphVizDiagram } from '@site/src/components/elements';

// Theme Components
import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';
import CodeBlock from '@theme/CodeBlock';
```

### Raw File Imports (Always use for code examples):
```jsx
import ExampleCode from '!!raw-loader!./assets/<mdx_filename>/example.js';
import ConfigCode from '!!raw-loader!./assets/<mdx_filename>/config.yaml';
import DiagramSource from '!!raw-loader!./assets/<mdx_filename>/diagram.mermaid';
```

### Default Front Matter (Always include at top of every MDX file):
```yaml
---
id: unique-doc-id
title: Page Title
description: Page description for SEO and metadata
sidebar_position: 1
---
```

**Important Rules:**
- Always import all modules even if not used
- Always use raw files for code examples  
- Never write code directly in MDX as it causes errors
- Store all code examples in assets folders
